# 子任务入口 UI 失效诊断

> 日期：2026-07-15
> 开发分支：`feat/checklist-subtasks`（已合入本地 `main`）
> 状态：根因已确认；入口已重构为不占布局的行内操作，用户已确认当前极简视觉方案；完整七态、打包和跨平台验收仍待完成

## 用户可见现象

- 扁平清单中每条普通任务下方都出现异常的大块空白，默认密度被破坏。
- “＋ 添加子任务”只在任务悬停时出现；用户从任务文字移向入口时，入口会消失，无法可靠判断或完成点击。
- 创建子任务后的层级只有缩进差异，控件、字号、分隔线和行高几乎与父任务相同，整体不像有意设计的层级。

## 根因

### 隐藏入口仍占布局

公共 `.checklist-add` 样式声明了 `min-height: 24px`，子任务入口 `.checklist-add--subtask` 只声明 `max-height: 0`，没有重置 `min-height`。当最小高度大于最大高度时，浏览器最终仍按最小高度布局。

用 Chromium 对同一组约束读取计算样式，结果为：

```json
{"minHeight":"24px","maxHeight":"0px","height":"24px","offsetHeight":24}
```

因此入口即使 `opacity: 0`，仍保留 24px 高度并参与 grid 布局；同时透明按钮仍是实际控件和潜在点击区域。这正是截图中任务间距被成倍拉大的直接原因。

### 悬停区域与可点击区域曾经分离

初版入口由任务行的 hover 控制，但入口位于任务行之外。鼠标离开任务行、尚未到达按钮时，hover 条件已经失效，按钮先消失。

当时的工作区已把任务行和按钮包进 `.checklist-parent`，并改由父容器的 `:hover` / `:focus-within` 控制；该轮检查时尚未经过真实指针轨迹验证，不能把源码结构标为已验证。

### 视觉方案未经状态设计

实现直接把规格中的文字示意翻译成“每个父任务下面再放一整行文字按钮”，没有先确认静止、悬停、编辑、键盘聚焦、已创建子任务和最窄窗口这几种状态。结果是入口出现时会让列表重排，子任务也缺少足够的视觉层级。这不是单个 CSS 数值可以解决的问题。

## 为什么自动测试和 review 没有发现

- `tests/checklist-subtasks-wiring.test.ts` 读取源码并用正则确认存在 `max-height: 0`、`:hover` 和 `:focus-within`，没有加载页面，也没有检查浏览器最终计算样式。该轮测试当时 6/6 通过，但不能发现 `min-height: 24px` 覆盖 `max-height: 0`。
- 现有测试没有覆盖元素几何尺寸、透明控件命中区域、鼠标从任务到入口的轨迹、布局截图或视觉回归。
- 早期两轮 review 都只检查代码和自动测试，没有启动开发应用执行 UI 验收，因此当时没有把规格中的“隐藏时不占空间”和“入口可达”逐条转成真实操作证据。
- 子任务改动同时包含数据模型、归一化、存储、键盘编辑、DOM 和 CSS，逻辑测试数量形成了错误的安全感，UI 风险没有作为独立 review 面处理。

## 后续闸门

1. 先重新确定不引发布局重排的鼠标入口方案，再修改 CSS。
2. 自动检查至少验证隐藏态的最终几何尺寸和命中区域，不能只匹配源码字符串。
3. 在开发应用中分别验证静止、悬停、从文字移动到入口、点击、键盘聚焦、父子层级和最窄窗口。
4. 视觉验证通过后，再对同一份最新 diff 完成 Codex 本地 review 与独立只读 review。

## 2026-07-15 入口重构复核

- 将“＋ 添加子任务”从父任务下方的独立文字行移到父任务行右侧的轻量 Plus 按钮，使用绝对定位，避免悬停时发生列表重排。
- 隐藏动作容器设置 `pointer-events: none`，只有悬停、编辑或键盘聚焦时恢复按钮命中，避免透明层挡住任务文字右侧点击。
- 父任务输入为动作按钮预留右侧内边距，并提高动作容器层级，避免长文本被 Plus / 删除按钮遮挡。
- 将动作容器放在文本框之前的 DOM 顺序，使键盘用户可以先 Tab 到入口；文本框自身的 Tab / Shift+Tab 缩进快捷键仍保持不变。
- 自动验证：471/471 测试、build、audit、diff check 通过。
- 开发窗口检查：普通清单不再出现入口撑出的额外空白；当前极简方案保留父任务右侧 Plus、父子缩进与浅色引导线，移除行内快捷键长文案；用户已确认该视觉方向。真实七态轨迹、打包应用和跨平台验收仍待完成。
