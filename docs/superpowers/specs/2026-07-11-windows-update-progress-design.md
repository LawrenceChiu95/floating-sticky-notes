# Windows 自动更新下载进度窗口设计

## 背景

Windows 用户确认“下载更新”后，现有确认框立即关闭，直到下载完成前都没有持续可见的反馈。用户容易误以为下载没有开始或应用失去响应。

本设计处理 [Issue #1](https://github.com/LawrenceChiu95/floating-sticky-notes/issues/1)：为 Windows 自动更新增加小型、非模态的下载进度窗口，同时保持便签可用，并继续复用现有 `electron-updater` 更新源和安装流程。

## 目标

- 用户确认下载后立即看到持续的下载状态。
- 有合法百分比时显示确定进度；没有百分比时显示明确的不定进度。
- 下载期间不重复检查或重复下载。
- 下载失败后清理进度状态，并显示可理解的错误。
- 下载完成后继续使用现有“重启并安装 / 稍后”确认流程。
- 所有便签关闭、应用仅托盘常驻时，更新流程仍然安全完成。
- 更新和安装继续保留 `%APPDATA%\floating-sticky-notes` 中的本地数据。

## 非目标

- 不改变公开通用更新源、更新元数据或 NSIS 安装方式。
- 不改变 macOS 的半自动 DMG 更新流程。
- 不提供暂停或取消下载；`electron-updater` 当前流程没有对应的可靠能力。
- 不在进度窗口中增加发布说明、设置或其他更新管理功能。

## 已选方案

新增独立、ownerless、非模态的轻量 `BrowserWindow`，由专门的进度窗口适配层管理。更新控制器只通过窄接口发出“打开、更新、完成、失败、恢复”指令，不直接操作 Electron 窗口。

不复用便签 React 页面或便签 preload。进度窗口使用独立的本地 HTML、独立最小 preload 和只读进度订阅，避免获得创建、删除或修改便签的 IPC 权限。

仅使用任务栏进度不足以满足 Issue 对持续可见窗口的要求，因此任务栏进度只作为进度窗口的补充和窗口故障时的降级反馈。

## 窗口行为

- 用户点击“下载更新”后立即创建并显示窗口，然后才调用 `downloadUpdate()`。
- 窗口独立于便签，不设置 parent，也不使用 modal。
- 窗口有普通标题栏，可拖动、可最小化、保留任务栏入口，但下载期间不可由用户关闭。
- 窗口创建时显示并聚焦；不保持永久全局置顶，避免慢速下载时持续遮挡其他应用。
- 窗口默认位于当前活动显示器的工作区中央，并限制在可见范围内。
- 应用明确退出、执行安装或系统结束会话时，程序可以强制销毁窗口；用户关闭限制不能阻止退出或关机。
- 窗口管理器持有唯一强引用。旧窗口的 `closed` 事件只有在引用仍指向该实例时才能清空引用，避免竞态误清理新窗口。
- 窗口创建或页面加载失败只记录错误并降级到任务栏进度，不中断实际下载。

## 进度数据流

1. 用户确认下载后，控制器进入 `downloading`，窗口显示“正在准备下载…”和不定进度。
2. `download-progress` 只在当前 operation 处于 `downloading` 时接受。
3. `percent` 必须是有限数值，并限制在 `0..100`。合法值显示四舍五入百分比和确定进度；缺失或非法值保持不定进度，不显示 `NaN%`。
4. 主进程保存当前 operation 的最新进度快照。renderer 尚未 ready 时到达的进度不会丢失；ready 后先发送最新快照，再继续推送后续变化。
5. 同一 operation 不会重复创建窗口或重置已知进度。托盘再次点击“检查更新”时恢复并聚焦现有进度窗，不发起新请求。
6. Windows 任务栏同步显示确定或不定进度；完成、失败和退出时清除。

## 更新状态机与竞态保护

控制器使用以下明确阶段：

```text
idle
  -> checking
  -> awaiting-download-confirmation
  -> downloading
  -> awaiting-install-confirmation
  -> downloaded-deferred
  -> installing
```

失败可以从检查、下载确认、下载或安装准备阶段回到 `idle`。每轮检查分配递增的 operation ID；所有 dialog Promise continuation 和 updater 事件都必须同时匹配 operation ID 与预期阶段后才能改变状态。

- `downloadUpdate()` Promise resolve 不代表完成；仅 `update-downloaded` 事件进入安装确认阶段。
- `downloadUpdate()` rejection 和 updater `error` 统一进入一次性失败终结函数，同一 operation 只清理、记录和提示一次。
- `download-progress`、`update-available`、`update-not-available` 和 `update-downloaded` 的迟到或不合阶段事件被忽略并记录。
- 重复 `update-downloaded` 只产生一个安装确认框。
- 错误发生后，旧的下载确认框即使稍后返回“下载更新”，也不能重新启动下载。
- 检查失败与下载失败使用不同标题，但都保持面向用户的简洁提示。
- 选择“稍后”进入 `downloaded-deferred`，不再次下载；本次会话后续手动检查应提示更新已经下载，而不是重新走检查。

## Windows 托盘生命周期修正

项目架构文档要求 Windows 关闭最后一张便签后仍由托盘常驻，但当前实现会在 `window-all-closed` 时退出。该行为与文档和 Issue 验收冲突，并会与 `autoInstallOnAppQuit = true` 组合，导致关闭进度窗时绕过安装确认。

本次一并修正为：

- Windows 和 macOS 在所有窗口关闭后继续运行；Linux 保持现有退出行为。
- Windows 仅在托盘“退出”、明确执行安装或系统退出时结束进程。
- 下载完成或失败时可以安全销毁进度窗，即使此时没有便签窗口，也不会触发意外退出或自动安装。
- 保留现有 `autoInstallOnAppQuit` 策略；“稍后”之后应用继续运行，用户后续通过托盘明确退出时由 updater 按现有策略安装已下载更新。

## 安全边界

- 进度窗口只加载打包内本地文件。
- `contextIsolation: true`、`nodeIntegration: false`、`sandbox: true`。
- 独立 preload 只暴露一个只读进度订阅，并提供 unsubscribe。
- preload 和 renderer 校验进度 payload；监听器只注册一次并在销毁时清理。
- 禁止进度窗口导航、创建新窗口或加载外部内容。
- 应用进入退出流程后，不再创建进度窗口、发送新 IPC 或弹出更新错误框。

## 错误与清理顺序

### 下载失败

1. 校验 operation 和阶段，并原子地标记该 operation 已终结。
2. 清除任务栏进度并允许程序关闭进度窗口。
3. 销毁进度窗口并释放监听器与引用。
4. 将控制器复位到 `idle`。
5. 若应用不在退出流程中，显示“下载更新失败”提示。

### 下载完成

1. 进入 `awaiting-install-confirmation`，停止接受进度事件。
2. 清除任务栏进度并关闭进度窗口。
3. 显示现有“重启并安装 / 稍后”确认。
4. “重启并安装”先保存所有便签，再进入 `installing` 并调用 `quitAndInstall(true, true)`。
5. “稍后”进入 `downloaded-deferred`，应用和托盘继续运行。

### 应用退出

- 标记 quitting，禁止后续 UI 创建与提示。
- 允许程序关闭或销毁进度窗口。
- 清除窗口引用、IPC 监听和任务栏进度。
- 继续使用现有退出前便签持久化流程。

## 自动化测试

### 更新控制器

- 确认下载后先打开不定进度，再调用 `downloadUpdate()`。
- 下载阶段重复手动检查不调用 `checkForUpdates()`，而是恢复已有进度窗口。
- error 后旧确认框返回不能启动下载。
- Promise rejection 与 updater `error` 双发只清理和提示一次。
- 迟到、越界、非法和重复 progress 不破坏当前状态。
- 重复 `update-downloaded` 只弹一次安装确认。
- 下载完成只以 `update-downloaded` 事件为准。
- “稍后”保留 `downloaded-deferred`，不重复检查或下载。

### 进度窗口管理器

- renderer ready 前收到 progress，ready 后仍收到最新快照。
- 合法百分比被限制并显示；非法值保持不定状态。
- 用户关闭被阻止，程序关闭、退出和销毁可以完成。
- 重复 open 复用并恢复现有窗口。
- 旧窗口 `closed` 不会清空新窗口引用。
- 创建或加载失败时清理引用、记录错误，且不会抛回更新控制器。
- 导航、`window.open` 和多余 IPC 能力被阻止。

### 应用生命周期与接线

- Windows 和 macOS 关闭全部窗口后保持运行，Linux 退出。
- 下载期间关闭全部便签不会退出应用。
- 无便签窗口时下载完成仍先显示安装确认；选择“稍后”不会立即安装。
- 退出时清理进度窗口和任务栏状态。
- macOS 更新路径不创建 Windows 进度窗口，现有测试继续通过。

## Windows 真机验收

使用足够慢的测试下载或限速代理，避免安装包瞬间完成而无法观察：

- 从旧版打包应用检查并确认下载，窗口立即出现不定进度，随后连续显示百分比。
- 下载期间便签可继续编辑、创建、关闭；关闭全部便签后托盘和进度仍存活。
- 进度窗可拖动、最小化，并可从任务栏恢复；用户不能关闭它。
- 覆盖多显示器、不同 DPI 和任务栏位置，窗口首次出现时始终可见。
- 下载期间断网，只出现一次错误提示，窗口和任务栏进度均清理，可再次检查。
- 下载期间从托盘重复检查，只恢复已有窗口，不重复请求。
- 无便签窗口时下载完成，仍出现“重启并安装 / 稍后”；选择“稍后”不退出、不安装。
- 选择“重启并安装”前保存待落盘内容，安装完成后版本正确，本地便签和图片保留。
- 下载中通过托盘退出以及 Windows 结束会话时，应用可以正常退出，不被不可关闭窗口卡住。
