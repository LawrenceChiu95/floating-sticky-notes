# 版本更新反馈与当前版本入口 Product Spec

> 日期：2026-07-15
> 独立评审：2026-07-16 已完成
> 状态：产品方案与内容规则已确认，待实现
> 目标版本：0.1.14
> 关联事项：[#4 版本更新反馈与当前版本入口](https://github.com/LawrenceChiu95/floating-sticky-notes/issues/4)

## 一句话决策

用户升级后第一次成功启动新版本时，展示当前版本的完整更新内容；托盘长期提供一个可点击的“版本 0.1.14”入口，用于确认当前版本并重新查看相同内容。更新内容只维护在 `CHANGELOG.md`，不建立第二份人工文案。

## 用户问题

当前更新流程只告诉用户发现、下载和安装了新版本，没有在安装成功后说明新版本带来了什么。用户经历了下载和重启，却缺少明确的交付反馈，也无法在应用内确认当前版本。

这带来两个直接问题：

1. 用户只感受到更新成本，不知道自己获得了什么。
2. 用户之后无法方便地确认当前安装版本或重新查看本版变化。

## 产品目标

- 更新真正完成后，用户能明确看到当前版本及本版变化。
- 同一版本的自动提示只出现一次，不在每次启动时打扰。
- 当前版本在托盘中长期可见，并可点击重新查看。
- 仓库、发布说明和应用内展示使用同一份版本事实。
- 更新内容随安装包提供，离线也能查看。
- 不让版本信息侵入每张便签的日常界面。

## 非目标

- 不新增独立“关于”窗口或富文本更新中心。
- 不浏览全部历史版本；第一版只查看当前安装版本。
- 不为应用内展示另写一份 release notes。
- 不运行时联网读取 GitHub Release 或其他远程内容。
- 不增加“查看完整日志”外链。
- 不改变 Windows 自动更新与 macOS 半自动 DMG 更新流程。
- 不把更新说明出现视为安装、差分下载或目标平台验证成功的证据。

## 内容单一真源

### 规则

`CHANGELOG.md` 是版本变化的唯一人工维护内容源，同时服务于：

- 仓库中的版本历史；
- GitHub Release 说明的内容基础；
- 应用内当前版本更新内容。

应用不维护 `release-notes.ts` 等第二份手写文案。

### 写作标准

每条用户可见变化都应：

- 先说明用户实际获得的能力或体验改善；
- 使用普通用户可以理解的语言；
- 一条只表达一个主要变化；
- 避免内部模块名、数据结构、provider、IPC 等实现细节；
- 内部重构若没有用户可见变化，不进入应用内更新内容。

必要的技术细节继续放在 commit、PR、设计文档或 `debug-log/` 中。

### 0.1.14 当前内容

准备发布时，现有“未发布”内容会归档为 `0.1.14`：

```markdown
## [0.1.14] - YYYY-MM-DD

### 新增

- 待办清单支持一层可选子任务，可用 Tab / Shift+Tab 调整层级，也提供“＋ 添加子任务”入口；父子任务可独立完成并持久化保存。

### 修复

- 待办事项现在会根据便签宽度完整换行并自动调整高度，长网址和连续字符不再被截断；中文输入法确认候选词时也不会误创建下一条待办。

### 变更

- 托盘现在会显示当前版本，点击即可随时查看本版新增与修复。
```

版本更新反馈能力使用上面的独立用户价值表述。应用内原样展示当前版本的全部用户可见条目，不支持“仅 Release 可见”或“仅应用内隐藏”的排除标记。这样既避免“现在会展示本版更新内容”一类自我指涉，也不增加第二套内容元数据。

## 交互设计

### 升级后首次展示

老用户升级后第一次成功启动新版本时，自动显示一次：

```text
本版更新

悬浮便签已更新到 v0.1.14

新增
• 待办清单支持一层可选子任务……

修复
• 窄便签中的待办文字现在可以完整换行……

[知道了]
```

展示内容为稳定核心版本章节（例如 `0.1.14-rc.1` 使用 `[0.1.14]`）的分类和条目，不是整个历史 CHANGELOG，也不合并跳过版本的日志。

“成功启动”至少满足：

1. Electron 主进程已经 ready；
2. 本地便签数据已经读取；
3. 便签窗口已经恢复或创建。

因此，不在下载完成或安装程序退出时提前显示“已更新”。

### 托盘入口

托盘菜单保持紧凑，只增加一个菜单项：

```text
新建便签
显示所有便签
开机时启动
检查更新
版本 0.1.14
────────────
退出
```

- 文案格式：`版本 {version}`，不带 `v`。
- 该项可以点击。
- 点击后展示当前安装版本的更新内容。
- 弹窗中的版本写作 `v0.1.14`，完整主文案为“悬浮便签已更新到 v0.1.14”或手动查看语境下的等价文案。
- 不再增加独立的“查看本版更新”或“关于悬浮便签”，避免托盘过长。

### 自动与手动路径

自动展示与托盘点击复用同一份版本内容和同一呈现组件，但语义有所区别：

- 自动路径：表达“已更新到”，展示成功后记录该版本已看过。
- 手动路径：表达当前正在使用的版本，任何时候均可打开，不改变已读状态。

## 展示规则

| 场景 | 自动展示 |
| --- | --- |
| 已安装 0.1.13，升级到 0.1.14 | 是，仅首次成功启动 |
| 再次启动 0.1.14 | 否 |
| 全新安装 0.1.14 | 否 |
| 0.1.14 升级到 0.1.15 | 是，仅首次成功启动 |
| 0.1.12 直接升级到 0.1.14 | 是，只展示 0.1.14 |
| 从较高版本降级到较低版本 | 否 |
| 开发环境启动 | 不自动展示 |
| 托盘点击“版本 0.1.14” | 始终允许 |

### 全新安装

全新安装没有经历更新，因此不自动展示“已更新”。第一次启动时将当前版本记为基线，避免第二次启动因为已经产生 `notes.json` 而被误判为升级。

### 首版能力自举

`0.1.14` 是第一个具备该能力的版本。由 `0.1.13` 升级的用户没有既有的“已看版本”状态，所以必须在便签初始化创建新文件之前判断 userData 中是否已有旧安装痕迹。

可用于判断既有安装的痕迹：

- 已存在 `notes.json`；或
- 已存在 `.auto-launch-default-applied`。

有旧安装痕迹且没有版本状态时，视为老用户升级；没有任何痕迹时，视为全新安装。

## 本地状态

在现有 userData 目录中增加独立状态文件，例如：

```json
{
  "lastShownReleaseVersion": "0.1.14"
}
```

约束：

- 不修改 `notes.json`，不让版本提示与便签数据耦合。
- 只接受合法语义化版本号。
- 当前版本高于已看版本时才自动展示。
- 降级时不把状态覆盖成更低版本。
- 文件缺失、损坏或字段非法时安全降级，不阻止便签启动。
- 状态写入失败不阻止应用运行；允许最坏情况下下次重复提示一次。

## 构建与发布内容生成

应用运行时不直接解析仓库中的 Markdown。构建前从 `CHANGELOG.md` 提取当前安装版本对应的稳定版本章节，生成只读的结构化构建产物，再随主进程 bundle 打入应用。

```text
CHANGELOG.md
    --> 构建时校验当前版本章节
    --> 生成当前版本的结构化只读数据
    --> 打入应用
    --> 自动展示与托盘手动查看
```

版本映射规则：

- 正式版本（例如 `0.1.14`）必须与 `[0.1.14]` 章节精确匹配；
- 预发布版本（例如 `0.1.14-rc.1`）从稳定核心版本 `[0.1.14]` 提取内容；
- 托盘与对话框始终显示完整安装版本，例如 `0.1.14-rc.1`；
- 预发布版本不自动展示，也不读取或修改正式版本的已读状态；
- 正式版或预发布版缺少对应稳定版本章节时，构建均失败。

该生成文件不是第二份人工真源：

- 不手写、不提交人工修改；
- 每次构建由 CHANGELOG 自动产生；
- 对应稳定版本缺少章节、章节重复或结构无法解析时，构建失败；
- 分类顺序和条目顺序与 CHANGELOG 保持一致；
- 应用离线运行时仍可查看。

预发布版本可从托盘手动验证内容，但不自动展示，也不修改正式用户的已读状态。

## 呈现方式

第一版优先使用 Electron 原生信息对话框：

- 复用项目现有对话框能力；
- 不新增 BrowserWindow、preload、renderer、CSP 和导航边界；
- 对当前 2～5 条分类内容足够；
- 自动与手动路径共用同一控制器，避免重复弹窗。

只有未来单版本内容明显超过原生对话框可读范围、需要图片或链接时，才重新评估专用窗口。

## 异常与并发

- 当前版本没有可展示内容：托盘点击时显示“本版本暂无更新说明”；自动路径不阻止启动。
- 状态文件读取失败：记录 warning，按安全规则降级。
- 状态写入失败：记录 warning，不阻止关闭对话框和继续使用。
- 启动时先完成本版更新提示，再执行静默检查更新，避免与“发现新版本”同时弹出。
- 连续点击版本菜单时只保留一个展示流程。
- 应用开始退出后不再创建更新说明对话框。
- 第二实例只恢复便签，不再次触发自动展示。

## 自动化验收

- CHANGELOG 当前正式版本章节可以稳定提取，分类和条目顺序不变。
- 正式构建在当前 package 版本缺少 CHANGELOG 章节时失败。
- 老用户首次升级展示一次，同版重启不再展示。
- 全新安装首启不展示，并建立当前版本基线。
- 跳版本只展示当前版本；降级不展示。
- 开发环境不自动展示，托盘手动入口可验证。
- 托盘只新增一个“版本 0.1.14”菜单项，点击回调正确。
- 自动与手动路径使用同一内容，手动查看不修改已读状态。
- 状态文件缺失、损坏、非法或写入失败不影响便签启动。
- 现有新建、恢复、开机启动、检查更新和退出行为不回归。

## 人工与产物验收

### 开发环境

- 托盘显示正确版本，长度和布局自然。
- 点击版本可以查看当前版本内容。
- 开发启动不会自动弹出更新提示。

### 打包应用

- 从 0.1.13 升级到 0.1.14 后只自动展示一次。
- 再次启动不重复展示，托盘仍可重新查看。
- 全新安装第一次和第二次启动都不误弹。
- Windows 和 Apple Silicon macOS 分别验证弹窗文案、托盘入口和状态持久化。
- 本地便签和图片不因状态文件或版本判断受到影响。

### 发布状态边界

- spec 确认不等于实现完成。
- 实现与自动测试通过不等于打包应用可用。
- 打包应用验证不等于 Windows/macOS 目标环境均通过。
- 更新说明正确展示不等于 Windows 差分下载链通过。
- 只有源码 Release、安装资源和更新元数据正式发布后，才能标记为 `released`；发布前必须另行确认。

## 成功标准

1. 用户升级后能立即知道本版新增或改善了什么。
2. 用户从托盘一眼能确认当前版本，并能点击重新查看。
3. 同一版本不会在每次启动时重复打扰。
4. CHANGELOG、Release 和应用内说明不会产生两份人工维护内容。
5. 版本反馈能力不污染便签界面，也不增加不必要的窗口与权限边界。
