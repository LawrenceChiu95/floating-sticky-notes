# 版本更新反馈与当前版本入口 Product Spec

> 日期：2026-07-15
> 自有窗口设计变更：2026-07-16
> 独立评审：2026-07-16 已完成；自有窗口细节已与 GLM 只读评审
> 状态：0.1.14 基础能力已发布；0.1.15 自有更新说明窗口正在发布验收
> 基线版本：0.1.14
> 目标版本：0.1.15
> 关联事项：[#4 版本更新反馈与当前版本入口](https://github.com/LawrenceChiu95/floating-sticky-notes/issues/4)

## 一句话决策

用户升级后第一次成功启动新版本时，在独立、非模态的更新说明窗口中展示从上次已展示版本到当前版本之间所有未读稳定版本的完整更新内容，并按版本分组；跳版本时只弹一个窗口，不让用户永久漏掉中间版本的用户可见变化。该窗口与下载进度窗口共享视觉语言，但为阅读单独设计。托盘长期提供一个可点击的版本入口，用于确认当前版本并重新查看当前安装版本内容。更新内容只维护在 `CHANGELOG.md`，不建立第二份人工文案。

## 用户问题

当前更新流程只告诉用户发现、下载和安装了新版本，没有在安装成功后说明新版本带来了什么。用户经历了下载和重启，却缺少明确的交付反馈，也无法在应用内确认当前版本。

这带来两个直接问题：

1. 用户只感受到更新成本，不知道自己获得了什么。
2. 用户之后无法方便地确认当前安装版本或重新查看本版变化。

## 产品目标

- 更新真正完成后，用户能明确看到当前版本及本版变化。
- 同一版本的自动提示只出现一次，不在每次启动时打扰。
- 当前版本在托盘中长期可见，并可点击重新查看。
- 仓库、发布说明和应用内展示使用同一份版本事实。
- 更新内容随安装包提供，离线也能查看。
- 不让版本信息侵入每张便签的日常界面。

## 非目标

- 不新增独立“关于”窗口或富文本更新中心；本条不排除本规格定义的更新说明阅读窗口。
- 不提供可主动浏览全部历史版本的更新中心；托盘手动路径只查看当前安装版本。自动升级路径仍按本规格一次性汇总未读版本。
- 不为应用内展示另写一份 release notes。
- 不运行时联网读取 GitHub Release 或其他远程内容。
- 不增加“查看完整日志”外链。
- 不改变 Windows 自动更新与 macOS 半自动 DMG 更新流程。
- 不在本轮重做“发现新版本”“更新已下载”或错误提示等其他更新对话框。
- 不复用下载进度窗口实例、preload、IPC channel、状态机或生命周期。
- 不把更新说明出现视为安装、差分下载或目标平台验证成功的证据。

## 内容单一真源

### 规则

`CHANGELOG.md` 是版本变化的唯一人工维护内容源，同时服务于：

- 仓库中的版本历史；
- GitHub Release 说明的内容基础；
- 应用内自动升级汇总与托盘当前版本说明。

应用不维护 `release-notes.ts` 等第二份手写文案。

### 写作标准

每条用户可见变化都应：

- 先说明用户实际获得的能力或体验改善；
- 使用普通用户可以理解的语言；
- 一条只表达一个主要变化；
- 不同功能点必须拆成独立的 `-` 无序列表项，不用分号把多项合并成一行；
- 避免内部模块名、数据结构、provider、IPC 等实现细节；
- 内部重构若没有用户可见变化，不进入应用内更新内容。

必要的技术细节继续放在 commit、PR、设计文档或 `debug-log/` 中。

### 0.1.14 已发布内容

`0.1.14` 的已发布内容以 [`CHANGELOG.md`](../../CHANGELOG.md) 中的同名章节为唯一真源。本规格不复制具体条目，避免分类或文案修正后与应用内展示漂移。

版本更新反馈能力使用上面的独立用户价值表述。应用内原样展示自动路径范围内各版本、或手动路径当前版本的全部用户可见条目，不支持“仅 Release 可见”或“仅应用内隐藏”的排除标记。这样既避免“现在会展示本版更新内容”一类自我指涉，也不增加第二套内容元数据。

## 交互设计

### 升级后首次展示

老用户升级后第一次成功启动新版本时，自动显示一次：

```text
本版更新

悬浮便签已更新到 v0.1.14

v0.1.14

新增
• 待办清单支持一层可选子任务……

修复
• 窄便签中的待办文字现在可以完整换行……

[知道了]
```

自动展示内容为从上次已展示稳定版本之后、直到当前稳定版本（含当前版本）的全部章节。多版本汇总按 SemVer 倒序展示，最新版本在最上方；每个版本内的分类和条目仍保持 `CHANGELOG.md` 顺序。例如已记录 `0.1.12` 为上次已展示版本，再升级到 `0.1.15` 时，在同一个窗口中依次展示 `0.1.15`、`0.1.14` 和 `0.1.13`，不创建三个窗口。旧安装没有任何已读基线时无法可靠反推出原安装版本，因此以完整优先：一次展示构建归档中不高于当前版本的全部可用稳定章节，允许迁移时重复已知内容，但不永久漏掉中间版本。预发布版本（例如 `0.1.15-rc.1`）仍使用稳定核心版本 `[0.1.15]` 的内容，但不自动展示。托盘手动查看保持当前安装版本内容，不把历史浏览入口扩展成新的“更新中心”。

“成功启动”至少满足：

1. Electron 主进程已经 ready；
2. 本地便签数据已经读取；
3. 便签窗口已经恢复或创建。

因此，不在下载完成或安装程序退出时提前显示“已更新”。

### 托盘入口

托盘菜单保持紧凑，只增加一个菜单项：

```text
新建便签
显示所有便签
开机时启动
检查更新
版本 0.1.14
────────────
退出
```

- 文案格式：`版本 {version}`，不带 `v`。
- 该项可以点击。
- 点击后展示当前安装版本的更新内容，不改变自动路径的已读状态。
- 窗口中的版本写作 `v0.1.14`，完整主文案为“悬浮便签已更新到 v0.1.14”或手动查看语境下的等价文案。
- 不再增加独立的“查看本版更新”或“关于悬浮便签”，避免托盘过长。

### 自动与手动路径

自动展示与托盘点击复用同一呈现组件，但内容范围和语义有所区别：

- 自动路径：表达“已更新到”，一次展示从上次已展示版本到当前版本的全部未读稳定章节，展示成功后记录当前最高版本已看过。
- 手动路径：表达当前正在使用的版本，只展示当前版本内容，任何时候均可打开，不改变已读状态。

## 展示规则

| 场景 | 自动展示 |
| --- | --- |
| 已安装 0.1.13，升级到 0.1.14 | 是，仅首次成功启动 |
| 再次启动 0.1.14 | 否 |
| 全新安装 0.1.14 | 否 |
| 0.1.14 升级到 0.1.15 | 是，一个窗口展示 0.1.15，仅首次成功启动 |
| 已记录 0.1.12，直接升级到 0.1.14 | 是，一个窗口展示 0.1.13 与 0.1.14 |
| 有旧安装痕迹但没有已读版本状态 | 是，一个窗口展示归档中不高于当前版本的全部稳定章节 |
| 从较高版本降级到较低版本 | 否 |
| 开发环境启动 | 不自动展示 |
| 托盘点击“版本 0.1.14” | 始终允许 |

### 全新安装

全新安装没有经历更新，因此不自动展示“已更新”。第一次启动时将当前版本记为基线，避免第二次启动因为已经产生 `notes.json` 而被误判为升级。

### 首版能力自举

`0.1.14` 是第一个具备该能力的版本。由 `0.1.13` 升级的用户没有既有的“已看版本”状态，所以必须在便签初始化创建新文件之前判断 userData 中是否已有旧安装痕迹。

可用于判断既有安装的痕迹：

- 已存在 `notes.json`；或
- 已存在 `.auto-launch-default-applied`。

有旧安装痕迹且没有版本状态时，视为老用户升级；没有任何痕迹时，视为全新安装。

## 本地状态

在现有 userData 目录中增加独立状态文件，例如：

```json
{
  "lastShownReleaseVersion": "0.1.14"
}
```

约束：

- 不修改 `notes.json`，不让版本提示与便签数据耦合。
- 只接受合法语义化版本号。
- 自动路径读取 `lastShownReleaseVersion`，收集其之后直到当前版本的全部稳定章节；当前版本高于已看版本时才自动展示。
- 没有已读状态但存在旧安装痕迹时，无法可靠推断原安装版本；为避免永久漏信息，展示构建归档中不高于当前版本的全部可用稳定章节，接受一次性重复；全新安装仍只建立基线，不自动展示。
- 降级时不把状态覆盖成更低版本。
- 文件缺失、损坏或字段非法时安全降级，不阻止便签启动。
- 状态写入失败不阻止应用运行；允许最坏情况下下次重复提示一次。

## 构建与发布内容生成

应用运行时不直接解析仓库中的 Markdown。构建前从 `CHANGELOG.md` 提取当前安装版本以及可用于补齐跳版本信息的稳定版本章节，生成只读的结构化历史窗口数据，再随主进程 bundle 打入应用。

```text
CHANGELOG.md
    --> 构建时校验稳定版本章节
    --> 生成按版本分组的结构化只读数据
    --> 打入应用
    --> 自动展示与托盘手动查看
```

版本映射规则：

- 正式版本（例如 `0.1.14`）必须与 `[0.1.14]` 章节精确匹配，并把可用稳定版本按语义化版本排序；
- 预发布版本（例如 `0.1.14-rc.1`）从稳定核心版本 `[0.1.14]` 提取内容，但不参与自动展示范围；
- 托盘与更新说明窗口始终显示完整安装版本，例如 `0.1.14-rc.1`；
- 预发布版本不自动展示，也不读取或修改正式版本的已读状态；
- 正式版或预发布版缺少对应稳定版本章节时，构建均失败；历史章节结构无法解析或重复时，构建也失败。

该生成文件不是第二份人工真源：

- 不手写、不提交人工修改；
- 每次构建由 CHANGELOG 自动产生；
- 对应稳定版本缺少章节、章节重复或结构无法解析时，构建失败；
- 归档版本按 SemVer 升序保存，每个版本内的分类和条目顺序与 CHANGELOG 保持一致；
- 稳定版本章节采用追加保留策略，不裁剪旧章节；这是无已读基线的旧安装避免永久漏信息的迁移保障；
- 应用离线运行时仍可查看。

预发布版本可从托盘手动验证内容，但不自动展示，也不修改正式用户的已读状态。

## 呈现方式

`0.1.14` 首版使用 Electron 原生信息对话框，已实现并随版本发布；发布前曾用功能分支构建版本号仍为 0.1.13 的本地验收包，完成 macOS 手动查看路径验证，自动首弹和状态持久化仍按发布监测记录。该本地验收包不等于公开的 0.1.13 Release，证据见 [Issue #4 macOS 人工验收评论](https://github.com/LawrenceChiu95/floating-sticky-notes/issues/4#issuecomment-4983625946)。由于系统默认视觉与现有下载进度窗口不一致，后续版本改用方向 C：独立、非模态、ownerless 的 `BrowserWindow`。它与下载进度窗口共享设计语言和安全壳范式，但布局为阅读更新说明单独设计。

### 视觉层级与文案

- 使用标准系统标题栏，窗口标题为“本版更新”，不在页面内伪造标题栏。
- 页面采用紧凑编辑式排版：顶行眉题自动路径显示“版本更新”、手动路径显示“当前版本”；右侧显示日期与条目数，跳版本汇总时改为版本数与总条目数。主标题 `<h1>` 只显示 `v{完整安装版本}`，使用数字衬线字体建立视觉焦点；中文继续使用共享系统字体栈。
- 自动路径的辅助文案为“悬浮便签已更新，以下是本次变化。”，手动路径为“以下是当前版本的新增与修复。”；文案保持通用，不为每个版本维护第二份手写摘要。
- 归档从 `CHANGELOG.md` 章节标题同步提取发布日期。单版本元信息显示 `YYYY.MM.DD · N 项`；多版本汇总显示 `N 个版本 · M 项`，每个版本组再显示自己的版本号与日期。预发布手动查看沿用稳定核心章节的发布日期。
- 单版本视图不重复显示可见版本组标题；多版本汇总使用 `<h2>` 区分版本，并按 SemVer 倒序把最新版本放在最上方。分类使用 `<h3>`、条目使用 `<ul><li>`；每个条目必须显示独立的可见圆点 marker，不得仅依赖 DOM 语义区分。每个版本内的分类和条目顺序与 `CHANGELOG.md` 一致。
- 每个分类使用左侧窄标签、右侧原文条目的双栏结构和细分隔线；不得根据标点自动拆分“标题/描述”，避免改变或误读 CHANGELOG 原文。
- 手动路径没有内容时显示“本版本暂无更新说明”，仍保留关闭入口。
- 与下载进度窗口共享米白背景 `#f5f1e8`、正文 `#26231f`、弱标题 `#5e574d`、次要文字 `#746b60`、琥珀强调色 `#d49a3a` 和字体栈。
- 页脚使用一个琥珀状态点；自动路径显示“已更新完成”，手动路径显示“当前已安装”。“知道了”使用琥珀色背景和正文色 `#26231f` 文字；不使用对比度不足的白色文字。

### 窗口与尺寸

- 窗口无 parent、`modal: false`、`alwaysOnTop: false`，用户查看更新说明时仍可编辑便签。
- 使用 `closable: true`、`minimizable: false`、`maximizable: false`、`resizable: false`、`skipTaskbar: false` 和 `autoHideMenuBar: true`。
- 标题栏关闭、页面“知道了”和 Escape 三个入口等价，都会销毁窗口；关闭不等于退出应用。
- 创建时按固定宽度 `min(440, workArea.width)` 和最小高度构建，避免随条目长度横向抖动；renderer 回传测量高度后，显示前只调整高度，不二次改变宽度。
- renderer 完成快照渲染并等待字体就绪后，回传内容高度。主进程把内容高度限制在 `180` 到 `max(180, min(560, floor(workArea.height * 0.75)))` 逻辑像素之间，并把标题栏等外层 chrome 计入最终窗口高度。
- 最终外层窗口宽高不得超过对应 workArea；若极小 workArea 无法容纳内容最小高度，则以可用 workArea 为准并保持内容区内部滚动。
- 少量内容按实际高度自然收紧；内容超过高度上限时，只让版本与更新条目区域内部滚动，编辑式页头和底部状态/“知道了”固定可见。不得通过缩小字号、压缩行高或折叠隐藏条目来容纳长内容。
- 滚动区域保留稳定滚动条空间，并在键盘聚焦时显示清晰轮廓；跳版本汇总仍只使用一个窗口，按版本倒序在同一滚动区连续展示。
- 窗口按鼠标所在显示器的工作区居中并限制在可见范围内。逻辑像素测量与 Electron 窗口尺寸保持同一口径，避免高 DPI 下重复换算。
- 创建时使用 `show: false` 和米白 `backgroundColor`。主进程收到 renderer 的渲染完成信号后先调整尺寸，再显示并聚焦，避免白闪和尺寸跳变。

### 显示、已读与单飞

- 自动和手动路径共用同一个 presenter 和唯一活动窗口；窗口已存在时只恢复并聚焦，不创建第二个窗口。
- 窗口创建时固定来源为 `automatic` 或 `manual`；后续聚焦不能改变来源或标题文案。
- renderer 的“渲染完成”信号只证明内容可测量，不等于用户已看到。主进程成功调用 `show()` 且窗口触发 `show` 后，才返回 `shown: true`。
- 只有来源为 `automatic` 的窗口真正显示成功后，才写入当前最高版本到 `lastShownReleaseVersion`。不等待用户关闭；标题栏关闭、“知道了”和 Escape 都不会再次改变已读状态。
- 手动窗口始终不写已读状态。自动请求若只复用了一个先前创建的手动窗口，也不得误写已读状态；presenter 必须返回活动窗口最初的来源。
- 自动路径等待 renderer 与窗口显示的总时限由具名常量 `RELEASE_FEEDBACK_RENDER_TIMEOUT_MS = 3000` 控制。超时、页面加载失败、preload 失效或显示失败时销毁隐藏窗口、返回 `shown: false`、记录 warning，并继续启动；本次不写已读状态，下次启动重试。
- `showAutomaticallyIfNeeded()` 只等待“显示成功或失败”的有界结果，不等待用户关闭。静默检查更新仍排在该结果之后，正常路径不再因更新说明窗口长时间阻塞。

### 无障碍与键盘

- 页面声明 `lang="zh-CN"`，使用语义化的 `<main>`、标题和列表结构。
- 内容渲染后焦点先落在可聚焦的主内容容器，屏幕阅读器先读主标题和条目；Tab 可以到达“知道了”，Enter 或 Space 关闭。
- Escape 通过最小 preload bridge 请求主进程关闭窗口。
- 内容溢出时，滚动区域支持 PageUp、PageDown、方向键和滚轮；焦点轮廓必须可见。
- 正文、次要文字、按钮文字和焦点状态均满足 WCAG AA 对比度。

### 安全和共享边界

- 窗口只加载打包内本地页面，继续使用 `contextIsolation: true`、`nodeIntegration: false`、`sandbox: true`。
- 使用独立的最小 preload，并构建为 CommonJS `.cjs`。它只暴露只读快照订阅和关闭请求，不获得便签读写、下载更新或任意 Node 权限。
- renderer 校验 `{ initiatedBy, version, releases }` 快照后才渲染，其中 `releases` 按版本包含各自的 `sections`；主进程校验 renderer 的来源窗口和高度数值后才调整窗口。
- 禁止页面导航、frame 导航和 `window.open`；CSP 至少限制 `default-src 'self'`、`script-src 'self'` 和 `style-src 'self'`。
- 下载进度窗口与更新说明窗口共享 CSS 设计 token、窗口安全选项范式和 manager 的生命周期模式；不共享窗口实例、manager 单例、IPC channel、preload、状态机、任务栏进度或关闭策略。
- 共享 token 使用单一 CSS 定义，两个窗口引用同一来源；manager 只按约定保持相似，不抽取耦合两个生命周期的公共基类。

## 异常与并发

- 当前版本没有可展示内容：自动路径不创建窗口并直接建立版本基线；托盘点击时在自有窗口中显示“本版本暂无更新说明”。自动路径有多个未读版本时，窗口按版本分组显示全部章节。
- 状态文件读取失败：记录 warning，按安全规则降级。
- 状态写入失败：记录 warning，不阻止关闭窗口和继续使用。
- 窗口创建、preload、renderer、内容渲染、尺寸测量或显示失败：不写已读状态，记录 warning，在 `RELEASE_FEEDBACK_RENDER_TIMEOUT_MS` 内释放隐藏窗口并继续启动；不回退到系统信息对话框。
- 启动时先等待本版更新窗口显示成功或失败，再执行静默检查更新；不等待用户关闭窗口。极低概率下，本版更新窗口可能与随后发现的新版本提示并存，本轮不为此重新引入模态阻塞。
- 连续点击版本菜单或自动/手动同时触发时只保留一个活动窗口；后续请求聚焦现有窗口。
- 应用开始退出后不再创建更新说明窗口，并销毁已经打开的窗口。
- 第二实例只恢复便签，不再次触发自动展示。

## 自动化验收

- CHANGELOG 当前正式版本和用于补齐跳版本信息的历史章节可以稳定提取；归档按 SemVer 升序保存，自动多版本展示按 SemVer 倒序输出，每个版本内的分类和条目顺序不变。
- 正式构建在当前 package 版本缺少 CHANGELOG 章节时失败。
- 老用户首次升级展示一次，同版重启不再展示。
- 全新安装首启不展示，并建立当前版本基线。
- 跳版本只弹一个窗口，并展示上次已展示版本之后直到当前版本的全部稳定章节；降级不展示。
- 开发环境不自动展示，托盘手动入口可验证。
- 托盘只保留一个版本菜单项，点击回调正确。
- 自动与手动路径使用同一呈现组件；自动展示未读版本汇总，手动展示当前版本；活动窗口单飞复用，重复触发只聚焦现有窗口。
- 活动窗口来源在创建时固定；手动窗口被自动请求复用时不得修改已读状态。
- renderer 完成渲染前不显示窗口；渲染完成后先按工作区限制尺寸，再显示并返回 `shown: true`。
- renderer 快照、内容高度、IPC 来源和关闭请求均经过校验；非法输入不改变窗口或已读状态。
- 页面加载失败、preload bridge 缺失、renderer 未回传、显示失败和 `RELEASE_FEEDBACK_RENDER_TIMEOUT_MS` 超时均返回 `shown: false`，不写已读状态且不会卡住启动；自动化测试使用 fake timer，不等待真实 3 秒。
- 标题栏关闭、“知道了”和 Escape 都销毁窗口；退出时销毁活动窗口且不再新建。
- 固定宽度、高度上下限、多显示器工作区限制和超限内部滚动均有纯逻辑或窗口 options 测试。
- 页面包含语义化标题与列表、可见焦点、键盘关闭和可滚动内容区；琥珀按钮使用满足 AA 的深色文字。
- renderer 的紧凑编辑式结构包含来源眉题、完整版本主标题、由 CHANGELOG 生成的日期/计数、分类双栏、固定页脚状态；单版本不重复可见版本标题，多版本仍有明确版本分组。
- 抽取共享 token 后，下载进度窗口的现有背景、字阶、进度条和提示层级保持视觉不变，并有视觉回归或稳定快照证据。
- 第三个 sandbox preload 构建为 CommonJS `.cjs`，主进程只加载 `.cjs`；renderer 多入口、CSP、导航守卫和 `window.open` deny 已接线。
- 状态文件缺失、损坏、非法或写入失败不影响便签启动。
- 自有窗口不获得便签读写、下载更新或其他多余 IPC 权限。
- 现有新建、恢复、开机启动、检查更新和退出行为不回归。

## 人工与产物验收

### 开发环境

- 托盘显示正确版本，长度和布局自然。
- 点击版本打开自有窗口，手动文案、分类、条目和版本正确。
- 窗口使用与下载进度一致的设计 token，但布局明显以阅读更新内容为主。
- 标题栏关闭、“知道了”和 Escape 均正常；连续点击只恢复同一个窗口。
- 条目少时窗口紧凑，条目多或跳版本汇总时高度不超过工作区上限并在内容区滚动；每个版本的标题和分类顺序清晰可辨。
- 长内容滚动时编辑式页头与底部“知道了”保持可见，不缩小正文、不折叠条目；Windows/macOS 的系统标题栏由各平台原生绘制，不显示伪造的交通灯或关闭按钮。
- 开发启动不会自动弹出更新提示。

### 打包应用

- 从上一正式版本升级到包含自有窗口的版本后，只自动展示一次。
- 再次启动不重复展示，托盘仍可重新查看。
- 全新安装第一次和第二次启动都不误弹。
- Windows 和 Apple Silicon macOS 分别验证窗口文案、托盘入口、显示后状态持久化、标题栏关闭、Escape、键盘焦点和长内容滚动。
- 在 Windows 多显示器和 100%/125%/150% DPI 下验证窗口始终可见、尺寸稳定、内容不被标题栏或任务栏裁切。
- 在 Apple Silicon macOS 上确认更新说明窗口不会意外新增 Dock 图标或进入应用切换器，同时窗口仍可通过托盘和窗口焦点正常访问。
- 破坏新 preload 或 renderer 页面时，自动路径不误记已读，启动不超过 `RELEASE_FEEDBACK_RENDER_TIMEOUT_MS` 等待并继续；修复后下次启动可重试。
- 打包后的 asar 必须包含独立更新说明页面和新的 `.cjs` preload；直接启动打包应用确认主进程、托盘、便签和两个更新窗口均可加载。
- 关闭更新说明窗口不能触发应用退出、更新安装或下载进度窗口清理。
- 本地便签和图片不因状态文件或版本判断受到影响。

### 发布状态边界

- spec 确认不等于实现完成。
- 实现与自动测试通过不等于打包应用可用。
- 打包应用验证不等于 Windows/macOS 目标环境均通过。
- 更新说明正确展示不等于 Windows 差分下载链通过。
- 只有源码 Release、安装资源和更新元数据正式发布后，才能标记为 `released`；发布前必须另行确认。

## 成功标准

1. 用户升级后能立即知道本次升级涉及的各版本新增或改善了什么。
2. 用户从托盘一眼能确认当前版本，并能点击重新查看。
3. 同一版本不会在每次启动时重复打扰。
4. CHANGELOG、Release 和应用内说明不会产生两份人工维护内容。
5. 版本反馈窗口与下载进度窗口属于同一视觉家族，同时保持独立、最小且可验证的权限与生命周期边界。
