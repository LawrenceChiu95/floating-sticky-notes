# 对话式项目管理协议

> 目标：用户只需要正常聊天和做产品判断；AI 负责把对话中的项目事实维护成下一次会话可以恢复的状态。

## 单一真源

| 事实 | 权威位置 | 其他位置的写法 |
| --- | --- | --- |
| 未完成的需求、Bug、反馈、研究和分发事项 | GitHub Issues | 只放 Issue 链接与必要摘要 |
| 当前目标、进行中事项、验证缺口和待同步队列 | 本机 `HANDOFF.md` | 不复制完整需求正文 |
| 已确认的产品方案与验收标准 | `docs/design/` | Issue 摘要并链接已公开的设计文档 |
| 已发布的用户可见变化 | `CHANGELOG.md` 与 GitHub Release | Issue 只补发布证据 |
| 长期架构和兼容边界 | `docs/architecture.md`、`docs/releasing.md`、`AGENTS.md` | 其他位置只引用 |
| 故障证据、根因与回归 | `debug-log/` | Issue 或 HANDOFF 只放指针 |

不要新增第二份任务清单或 metadata 状态库。`HANDOFF.md` 是便于冷启动的本机缓存，GitHub Issues 才是动态事项真源。

## 对话中的识别规则

### 自动收件

出现以下信息时，AI 应主动搜索现有 Issue，能合并就追加证据，否则建立最小事项：

- 用户或使用者遇到可复现问题；
- 用户明确说“记一下”“以后想做”“下个版本做”或形成了产品结论；
- 开发、测试或发布中发现新的独立风险或后续工作；
- 获得了实现、测试、打包、真机、发布或线上监测的新证据。

每个事项至少保留：来源、用户问题或目标、当前证据、最小成功行为、明确非目标（适用时）、风险/平台和下一步。不要把用户随口提出的解决方案直接当作问题定义。

### 先讨论，不急着建正式事项

纯提问、开放脑暴、技术可行性讨论或没有形成结论的一句灵感，默认留在当前对话。只有用户要求保留，或讨论已形成清晰问题与下一步时，才进入 Inbox 性质的 Issue。

不要因为 AI 能写出完整文档，就把低证据想法膨胀成已排期需求。

### 自动维护动作

无需询问用户即可：

1. 搜索并合并重复事项；
2. 在不泄露隐私的前提下补充反馈与验证证据；
3. 使用仓库已有标签做最小分类；
4. 把已交付但仍开放的 Issue 补证据后关闭；
5. 将暂时不做的事项标为 `暂不处理`，保留恢复条件；
6. 在 `HANDOFF.md` 中同步当前目标、活跃事项、验证缺口和待决策项；
7. 收尾时给用户一行简短回执，例如“已维护：反馈合并到 #12，新增 Windows 证据，暂不排期”。

不要为了管理而创建大量标签、字段、Milestone 或 Project。事项数量和协调成本明显上升后，再引入额外视图。

## 必须询问用户的边界

以下决定不能由 AI 代替：

- 改变产品定位、目标用户或长期维护承诺；
- 多个方向互斥，或优先级取舍会显著挤占其他方向；
- 无法可靠判断用户指的是哪个事项、版本或范围；
- 公开包含用户隐私、日志、截图或其他敏感信息；
- 发布、删除、迁移数据、改变安装身份等对外或难以撤销的操作。

普通 Issue 的创建、去重、补证据、分类和状态校正属于已授权的日常维护，不重复询问。

## 状态必须诚实

项目不使用单一的“完成”概括交付。按以下维度分别记录，并在 `verified` 或 `released` 后附证据：

| 维度 | 问题 |
| --- | --- |
| 实现 | 代码或配置是否已接线 |
| 自动验证 | 测试、构建和静态检查是否通过 |
| 产物验证 | 打包后的真实产物是否能启动和运行 |
| 目标环境验证 | 是否在真实平台、权限、网络和数据下验证 |
| 发布 | 用户是否已经可以获得变更 |
| 监测 | 发布后的真实路径是否稳定 |

推荐状态词：`not-started`、`in-progress`、`blocked`、`verified`、`not-applicable`；只有发布维度使用 `released`。

硬规则：

- 实现完成不等于自动验证通过；
- 自动验证通过不等于产物可用；
- 交叉构建不等于目标平台真机验证；
- 已合入不等于已发布；
- 没有新证据时，不能因为时间推移自动升级状态。

## 新会话启动协议

只要本轮涉及项目事项，在已读取本机 `HANDOFF.md`（若存在）的基础上：

1. 读取 `AGENTS.md` 与本协议；
2. 用 `HANDOFF.md` 恢复当前目标和未同步队列；
3. 查询相关 GitHub Issues，GitHub 状态覆盖 HANDOFF 中过期的摘要；
4. 按风险加载架构、发布、设计或诊断文档；
5. 开始工作前，确认没有把历史“已实现”误当成“已验证”或“已发布”。

无需每次向用户汇报完整恢复过程；只在发现冲突、阻塞或需要决定时说明。

## 每轮收尾协议

完成一段实质工作后，AI 必须检查：

1. 是否出现新的需求、Bug、反馈、风险或后续任务；
2. 相关 Issue 是否需要补证据、迁移状态、关闭或标记暂不处理；
3. 实现、自动验证、产物、真机、发布和监测状态是否分别准确；
4. `HANDOFF.md` 的当前目标、下一步和待决策项是否仍然有效；
5. 用户可见且已发布的变化是否需要进入 `CHANGELOG.md`；
6. 是否意外在多个文件复制了同一份活状态。

## GitHub 不可用时

不要假装已经完成同步。把最小记录暂存在 `HANDOFF.md` 的“待同步到 GitHub”章节，包含日期、事项摘要、证据和期望动作。下次权限恢复后：

1. 先搜索是否已有重复 Issue；
2. 同步或合并记录；
3. 从待同步队列删除已成功处理的条目；
4. 向用户如实回执同步结果。

`HANDOFF.md` 的待同步章节是失败队列，不是第二套长期任务系统，正常情况下应为空。
